# AllIn.LowCodeKit 项目计划

> 最后基于当前代码整理时间：2026-03-29

## 一、计划原则

- 当前计划必须以工作区里的现有代码实现为基线。
- 不再按早期 Phase 文档把项目视为“待从零实现”。
- 当前阶段更适合按“构建稳定性、入口收敛、主链路验收、遗留整理”来推进。

## 二、当前总目标

在现有大部分功能已经落地的基础上，把项目整理到以下状态：

- 前后端均可稳定构建
- 全局配置入口清晰且不再分叉
- 主业务链路可完整验证
- 高耦合页面逐步降复杂度
- 文档能稳定反映当前代码真相

## 三、当前最高优先级

### 1. 维持前后端构建绿灯

- 当前前后端构建都已经恢复通过
- 后续新增改动时，继续把 `dotnet build` 和 `npm run build` 作为默认验收动作
- 当前应避免再次引入破坏构建的类型错误或构建配置问题
- 对 Vue 组件改动，默认先跑 `npx.cmd vue-tsc -b`，再判断是代码问题还是编辑器扩展误报

### 2. 维持后端持续可构建

- 继续保持 `backend/` 可通过 `dotnet build`
- 后续只要改到动态数据、导入导出、自动化执行、全局配置持久化，至少跑一次后端构建

### 3. 以当前拆分后的全局配置入口为准

- 正式入口继续固定为：
  - `/global-config/automation`
  - `/global-config/system`
- `/global-config` 只作为兼容跳转层
- 后续需要明确 `frontend/src/views/GlobalConfigView.vue` 的命运：
  - 删除
  - 保留兼容
  - 仅保留历史参考

## 五、当前主要风险

### 1. 遗留入口风险

- `GlobalConfigView.vue` 仍存在，容易让后续修改误入旧入口
- 如果继续同时维护旧入口和新入口，配置逻辑会再次分叉

### 2. 前端耦合风险

- `ModuleView.vue` 承担了太多职责
- `moduleStore.ts` 汇总了大量主流程状态，改动时要注意跨功能回归
- 自动化页、系统配置页和遗留全局配置页之间存在重复逻辑风险

### 3. 开发环境一致性风险

- 如果开发机同时启用 `Vetur` 和 `Vue - Official`，`.vue` 文件可能出现大量误报
- 如果 VS Code 前置任务仍走不稳定的 shell 启动方式，可能再次出现“启动前端/后端存在错误”
- 如果以后新增 Vue 组件继续使用对旧扩展敏感的写法，可能再次触发编辑器误诊断

### 4. 构建回归风险

- 当前构建已恢复，但任何对大页面、大 Store 或构建配置的改动，都可能再次引入回归
- 由于页面与依赖较重，前端构建验证不能省略

### 5. 历史计划偏差风险

- 旧 Claude 计划中部分菜单结构、页面入口和阶段划分已经不适用于当前代码
- 如果按旧计划直接实施，容易做出与现状不一致的修改

## 六、建议的推进顺序

### 第一阶段：稳定性

1. 保持前后端构建持续通过
2. 对每次改动后的前端与后端分别做构建验证
3. Vue 组件新增或重构后，先执行 `npx.cmd vue-tsc -b`
4. 优先修掉新的构建回归，再继续叠加功能

### 第二阶段：开发环境收敛

1. 继续把 `Vue - Official` 作为唯一推荐 Vue 扩展
2. 保持工作区对 `Vetur` 误报的抑制配置
3. 保持 VS Code 任务使用显式 `npm.cmd` / `dotnet` 的启动方式
4. 新建 Vue SFC 时优先沿用当前已验证通过的宏写法与导入方式

### 第三阶段：入口收敛

1. 明确遗留 `GlobalConfigView.vue` 的保留策略
2. 清理或隔离与正式入口重复的配置逻辑
3. 确保后续任务统一以拆分页为准

### 第四阶段：链路验收

1. 菜单链路验收
2. 表单设计链路验收
3. 模块数据链路验收
4. Excel 导入导出链路验收
5. 自动化链路验收
6. 系统配置链路验收

### 第五阶段：结构整理

1. 继续拆分 `ModuleView.vue`
2. 抽取系统配置和自动化配置中的公共逻辑
3. 清理调试输出、重复逻辑和遗留页面依赖

## 七、默认验收标准

- 后端改动后执行：`dotnet build`
- 前端改动后执行：`npm run build`
- 对当前触达链路做最小闭环手工验证
- 如果修改了路由、全局配置或导入导出流程，需要额外核对页面跳转和状态恢复
